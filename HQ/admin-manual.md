## 0. はじめに

このマニュアルは ZeroEn の日常運営を担当するあなた向けの手引きです。

**ZeroEn とは何か：**
日本のクライアント向けにウェブサイトを無料で作り、月額サブスクリプション（¥5,000 または ¥10,000/月）で運営費用を回収するビジネス。タグライン：「アイデアを、形にする。」

**あなたの役割：**
- Coconala の受信箱を確認し、未返信を整理する
- 支払い状況を監視し、遅延に対応する
- クライアントのライフサイクル（問い合わせ → オンボーディング → 月次運用 → 解約）を管理する
- 大東がコード・デプロイ・レポート生成などの技術タスクに集中できるよう、それ以外の全コミュニケーションを処理する

### 2.1 新規リード受付

**Coconala 経由の場合：**

1. Coconala のメッセージで問い合わせを受信する
2. 24 時間以内にテンプレ 01（初回返信）で返信する
3. スコーピング質問（テンプレ 02）を送る → セクション 2.2 へ

**zeroen.dev 経由の場合：**

1. 大東に通知する（申し込みフォームの通知は大東が受け取る設定になっているはず）
2. 大東の指示を待つ

**エスカレ条件：**
- 「今すぐ依頼したい」「大きな予算がある」「株式の話をしたい」などの大型案件の雰囲気
- 日本語以外のクライアント
- 怒り口調・威圧的なトーン

---

### 2.2 スコーピング・ヒアリング（5問スコアリング）

以下の 5 つの質問を送る（テンプレ 02）。回答を受け取ったら採点して大東に共有する。

| # | 質問 | 合格基準 |
|---|------|---------|
| 1 | ウェブサイトの目的は何ですか？ | 明確な目的がある（「なんとなく欲しい」は NG） |
| 2 | ターゲットとなるお客様はどんな方ですか？ | 具体的に説明できる |
| 3 | 掲載するテキスト・画像などのコンテンツは 1 週間以内に提供できますか？ | 「はい」または明確な期日 |
| 4 | 6ヶ月の最低契約期間があることはご理解いただけますか？ | 明確に同意 |
| 5 | オンボーディング中、48 時間以内の返信は可能ですか？ | 「はい」 |

**採点：** 5 問すべて合格 → 大東に「受け入れ推奨」と報告。1 問でも失格 → 「要確認」と報告。大東が最終判断する。

### 2.3 オンボーディング・チェックリスト

大東から「このクライアント受け入れます」と連絡が来たら、以下を順番に処理する。

- [ ] Coconala で ¥500 の intake 支払いを確認する
- [ ] 6ヶ月コミットを**書面（Coconala メッセージ上）**で確認する → テンプレ 03
- [ ] クライアントの確認メッセージをスクショして保存する
- [ ] Stripe サブスクリプションの設定（大東が行う。完了の連絡を待つ）
- [ ] スコープ凍結の確認メッセージを送る → テンプレ 04
- [ ] `HQ/crm/clients.json` への登録（大東が行う）
- [ ] 大東に「オンボーディング完了」を報告する

**重要：** 6ヶ月コミットの書面確認が取れるまでビルドは開始しない。

### 2.7 追加リクエスト対応

クライアントから変更や追加の依頼が来たら、以下のフローで処理する。

#### STEP 1：スコープ判定

| 判定 | 条件 |
|------|------|
| **月次 included** | そのクライアントの今月の変更残数が残っている、かつ Small または Medium の範囲 |
| **per-request 請求** | 今月の変更残数を使い切った、または Large サイズ |
| **スコープ外・大東へ** | Phase 1 対象外の機能（ログイン・DB・EC等）、または判断に迷う |

#### STEP 2：サイズ判定（per-request の場合）

`HQ/crm/change-catalogue.md` の早見表を参照する。迷ったら大東に確認する。

| サイズ | 目安 | 価格 |
|--------|------|------|
| Small | 30 分以内の作業（テキスト修正、画像差し替え等） | ¥4,000 |
| Medium | 1〜3 時間の作業（新規セクション追加、フォーム設置等） | ¥10,000 |
| Large | 半日以上（機能追加、全面リデザイン等） | ¥25,000〜（個別見積もり） |

#### STEP 3：返信

- included の場合：テンプレ 07-A（無料対応の旨を伝える）
- per-request の場合：テンプレ 07-B（見積もりを送る）
- スコープ外の場合：大東に確認後、テンプレ 07-C

変更の実作業は常に大東が行う。あなたはコミュニケーションのみ担当する。

---

### 2.8 未払い対応

Stripe で支払い失敗を検知したら、以下のタイムラインで対応する。

| タイミング | アクション | 送るメッセージ |
|-----------|-----------|---------------|
| Day +1（失敗翌日） | リマインダー送信 | テンプレ 08-A |
| Day +7 | 再リマインダー送信 | テンプレ 08-B |
| Day +14 | 大東に連絡 → サイト一時停止（大東が実施）→ 停止通知送信 | テンプレ 08-C |
| Day +44 | 大東に連絡 → アーカイブ（大東が実施）→ アーカイブ通知送信 | テンプレ 08-D |

**重要：**
- Day +14 のサイト停止、Day +44 のアーカイブは**必ず大東に連絡してから**実施する。あなたは通知メッセージの送信のみ担当する。
- 90 日以内に未払い分を全額支払えば再アクティベート可能。

---

### 2.9 解約対応

クライアントから「解約したい」と連絡が来たら：

1. 解約を受け付けた旨と、ポリシーの確認メッセージを送る → テンプレ 09-A
2. 6ヶ月コミットの残期間を確認する（いつ契約開始したか → `HQ/crm/clients.json` を参照）
3. 以下の計算を大東に報告する：

**早期解約（6 ヶ月未満）の計算：**
- 残月数 × 月額 = 早期解約料
- 大東が最終確認

**6 ヶ月以上経過の場合：**
- 違約金なしで解約可能
- サイトはアーカイブ
- コードは ZeroEn が保持（引き渡しなし・ソースコードの譲渡は一切行わない）

4. 大東の確認を取ってから、金額確定の連絡を送る → テンプレ 09-B

### 3.4 ブランドボイス（日本語での文章作成ガイド）

ZeroEn のトーンを守って文章を書くこと。

**原則：**
- 短く、直接的に。余分な言葉を削る
- 自信があるが、傲慢ではない
- 具体的な数字・期日・条件を使う。曖昧な表現は避ける
- 「翻訳された英語」っぽくならない。自然な日本語で

**良い例 / 悪い例：**

| 良い | 悪い |
|------|------|
| 「ご連絡ありがとうございます。3営業日以内に対応します。」 | 「この度はお問い合わせいただきまして誠にありがとうございます。担当者よりご連絡申し上げます。」 |
| 「今月の変更枠は残り1回です。」 | 「現在のご契約プランにおける月次変更可能回数の残余数についてご案内いたします。」 |
| 「6ヶ月コミットが必要です。同意の上でお申し込みください。」 | 「弊社サービスをご利用いただくにあたり、一定期間のご契約が条件となっております。」 |

### 3.5 用語集

| 用語 | 意味（非エンジニア向け） |
|------|------------------------|
| Vercel | ZeroEn がクライアントのサイトを公開するためのサービス（サーバーの役割）。クライアントは直接触らない |
| Stripe | オンライン決済サービス。月額サブスクリプションの請求はここで管理する |
| Supabase | データベースサービス。Phase 1（現在）は使用しない |
| Coconala | 日本のフリーランスマーケットプレイス。ZeroEn の集客チャンネルのひとつ |
| サブスクリプション | 月額の定期支払い契約 |
| SAFE（セーフ） | Simple Agreement for Future Equity の略。将来の株式取得権を約束する契約書。法人化時に株式に転換する |
| レベニューシェア（売上分配） | クライアントの売上の一定割合（約 10%）を ZeroEn が受け取る契約 |
| DNS | ドメイン（例：example.com）とサーバーをつなぐ設定。大東が管理する |
| ドメイン | ウェブサイトのアドレス（例：yourshop.com）。クライアント自身が購入する |
| スコープ | 何を作るか・何をやるかの範囲。契約時に確定し、後から変更する場合は追加料金が発生する |
| 6ヶ月コミット | 契約開始から最低 6 ヶ月は解約できないという合意。無料ビルドの条件。早期解約は残月数分の請求となる |
| グレース期間 | 支払い期日を過ぎてから強制対応（停止・解約）が発動するまでの猶予期間 |
| Phase 1 | 現在の段階。ランディングページ・ホームページのみ対応。データベースや認証機能は対象外 |

## 4. セーフティ・レール

### 4.1 絶対にやってはいけないこと

| # | やってはいけないこと | 理由 |
|---|---------------------|------|
| 1 | 大東の確認なしにクライアントへ何かを送る（テンプレ以外） | 誤った情報・約束のリスク |
| 2 | スコープを勝手に変更・承認する | 無償労働・契約トラブルのリスク |
| 3 | API キー・パスワード・機密情報を誰かに共有する | セキュリティ事故のリスク |
| 4 | Coconala 経由のクライアントをプラットフォーム外（Stripe など）で請求する | Coconala 規約違反・アカウント停止のリスク |
| 5 | `Clients/` ディレクトリを開く・編集する | クライアントコードへの意図しない変更 |
| 6 | git コマンドを実行する | リポジトリ破壊のリスク |
| 7 | 6ヶ月コミットの書面確認前にビルド開始を伝える | 無償労働リスク |
| 8 | クライアントに解約金額を独断で伝える | 計算ミス・契約トラブルのリスク |
| 9 | Vercel / Stripe の設定を変更する | 本番サイトの障害リスク |
